## 1. 品質評估總覽

### 1.1 評估架構

每個 Skill 的品質由 **6 個維度** 組成，每個維度 0-100 分：

| 維度                       | 英文代號        | 主要評估內容                       |
|----------------------------|-----------------|------------------------------------|
| **任務適配度與問題定義**   | problemFit      | 目標清晰度、工作流閉環、輸入精準度 |
| **正確性與可驗證性**       | correctness     | 方法論可重現、腳本吻合、範例可回歸 |
| **資料來源品質與資料治理** | dataGovernance  | 來源完整性、可追溯性、交叉驗證     |
| **穩健性與容錯**           | robustness      | 失敗模式處理、降級策略、錯誤回報   |
| **可重現性與可維護性**     | maintainability | 版本管理、模板穩定、避免文件漂移   |
| **輸出可用性與決策支援**   | usability       | 結論明確、依據完整、歷史對照       |

### 1.2 總分計算

```
overall = (problemFit + correctness + dataGovernance + robustness + maintainability + usability) / 6
```

### 1.3 skill.yaml 配置結構

```yaml
qualityScore:
  overall: 65          # 整體分數（0-100）
  badge: 白銀          # 等級徽章
  metrics:
    problemFit: 70       # 任務適配度與問題定義
    correctness: 75      # 正確性與可驗證性
    dataGovernance: 65   # 資料來源品質與資料治理
    robustness: 60       # 穩健性與容錯
    maintainability: 70  # 可重現性與可維護性
    usability: 75        # 輸出可用性與決策支援
  details: |
    **任務適配度（70/100）**
    - 具體評語...
```

## 2. 六維度評分標準

---

### 2.1 任務適配度與問題定義（Problem Fit）

#### 主要看哪裡

| 檔案位置                     | 評估重點                                     |
|------------------------------|----------------------------------------------|
| `SKILL.md`                   | 一句話目標、使用情境、非目標（不做什麼）     |
| `workflows/*.md`             | 實際工作流是否能完成該目標（從輸入到輸出）   |
| `references/input-schema.md` | 輸入是否剛好支撐任務（不缺關鍵欄位也不過度） |

#### 檢查點

**A. SKILL.md 是否清楚寫出：**

| 檢查項           | 滿分標準                 | 扣分條件       |
|------------------|--------------------------|----------------|
| 使用者要解的問題 | 一句話說明核心問題       | 模糊或缺失 -10 |
| 輸出要支持的決策 | 明確說明輸出用於什麼決策 | 未說明 -10     |
| 適用情境         | 列出 2+ 個適用場景       | 缺失 -5        |
| 不適用情境       | 明確說明不做什麼         | 缺失 -5        |

**B. workflows/ 是否覆蓋：**

| 檢查項               | 滿分標準                           | 扣分條件      |
|----------------------|------------------------------------|---------------|
| 主路徑（happy path） | 有完整的 analyze.md 或等效主工作流 | 缺失 -15      |
| 缺資料分支           | 說明資料不全時的處理方式           | 缺失 -5       |
| 不同市場分支         | 若適用，覆蓋不同市場邏輯           | 該有但缺失 -5 |
| 不同頻率分支         | 若適用，覆蓋日/週/月不同頻率       | 該有但缺失 -5 |

**C. input-schema.md 是否避免「模糊欄位」：**

| 檢查項       | 滿分標準                        | 扣分條件         |
|--------------|---------------------------------|------------------|
| 欄位定義清晰 | 每個欄位有明確說明              | 模糊定義 -3/欄位 |
| 單位明確     | 噸/盎司/張等單位清楚            | 缺少單位 -3/欄位 |
| 合約類型明確 | spot/future/continuous 定義清楚 | 混淆 -5          |
| 時間口徑明確 | 日期格式、時區說明              | 缺失 -5          |

#### 評分等級

| 分數區間   | 等級   | 標準描述                                 |
|------------|--------|------------------------------------------|
| **90-100** | 優秀   | 問題定義精準，工作流完整閉環，輸入無歧義 |
| **80-89**  | 良好   | 問題定義清楚，主路徑完整，少數邊界未覆蓋 |
| **70-79**  | 中等   | 問題定義基本清楚，但分支覆蓋不足         |
| **60-69**  | 待改進 | 問題定義模糊，工作流有缺口               |
| **<60**    | 不合格 | 目標不明確，無法理解要解決什麼問題       |

#### 快速訊號（從目錄結構判斷）

```
✅ 高分訊號：
  - 有 workflows/analyze.md + templates/output-*：任務閉環較完整
  - SKILL.md 有明確的 <objective> 和 <success_criteria>
  - input-schema.md 每個欄位都有 example 和 unit

⚠️ 低分訊號：
  - 缺 workflows 或 templates：「定義有了，但落地不可用」
  - SKILL.md 只有描述，沒有具體的適用/不適用情境
  - input-schema.md 欄位名模糊（如 symbol 未說明是 spot 還是 future）
```

### 2.2 正確性與可驗證性（Correctness & Verifiability）

#### 主要看哪裡

| 檔案位置                                | 評估重點                                          |
|-----------------------------------------|---------------------------------------------------|
| `references/methodology.md`（或根目錄） | 公式、規則、假設                                  |
| `scripts/*.py`                          | 是否真的照方法論實作（threshold、視窗、資料處理） |
| `examples/*`                            | 是否提供可重跑案例（golden case）                 |

#### 檢查點

**A. methodology 是否能「逐步重現」：**

| 檢查項    | 滿分標準               | 扣分條件         |
|-----------|------------------------|------------------|
| 輸入定義  | 所有輸入變數有明確來源 | 缺失定義 -5/變數 |
| 中間特徵  | 中間計算步驟可追溯     | 黑盒步驟 -10/處  |
| 規則/模型 | 判斷規則或模型參數明確 | 參數來源不明 -10 |
| 輸出定義  | 輸出格式和含義清楚     | 輸出模糊 -5      |

**B. scripts/ 是否有：**

| 檢查項              | 滿分標準                              | 扣分條件           |
|---------------------|---------------------------------------|--------------------|
| 明確參數            | window、threshold、frequency 集中定義 | 分散或硬編碼 -5/處 |
| 單位與對齊          | 時區、交易日、rolling 對齊方式明確    | 未處理對齊 -10     |
| 最小可重現          | 固定 seed / 固定資料版本（若適用）    | 無法重現 -10       |
| 與 methodology 吻合 | 腳本邏輯與文檔一致                    | 不一致 -15         |

**C. examples/ 是否能做 regression test：**

| 檢查項     | 滿分標準                  | 扣分條件      |
|------------|---------------------------|---------------|
| 範例存在   | 至少 1 個完整 golden case | 無範例 -15    |
| 可重跑     | 同版本同輸入，結論一致    | 無法重跑 -10  |
| 差異可解釋 | 若有差異，有說明原因      | 差異無解釋 -5 |

#### 評分等級

| 分數區間   | 等級   | 標準描述                                                 |
|------------|--------|----------------------------------------------------------|
| **90-100** | 優秀   | 方法論完整可重現，腳本與文檔完全吻合，有多個 golden case |
| **80-89**  | 良好   | 方法論清晰，腳本基本吻合，有可用範例                     |
| **70-79**  | 中等   | 方法論大致可追溯，但部分細節缺失                         |
| **60-69**  | 待改進 | 方法論不完整，腳本與文檔有出入                           |
| **<60**    | 不合格 | 無法驗證正確性                                           |

#### 快速訊號

```
✅ 高分訊號：
  - 有 examples/：更容易做自動化正確性檢查
  - 額外拆出 regime-rules.md 之類文件：規則可獨立審核
  - methodology.md 有 LaTeX 公式或等效明確表示

⚠️ 低分訊號：
  - 只有 methodology.md 但無公式，只有文字描述
  - scripts/ 有 magic numbers 分散各處
  - examples/ 為空
```

### 2.3 資料來源品質與資料治理（Data Provenance & Governance）

#### 主要看哪裡

| 檔案位置                                       | 評估重點                                   |
|------------------------------------------------|--------------------------------------------|
| `references/data-sources.md`                   | 來源、欄位、頻率、延遲、授權、替代方案     |
| `scripts/fetch_data.py` 或 `fetch_*`           | 是否落實資料治理（cache、timestamp、版本） |
| `workflows/fetch-data.md`、`cross-validate.md` | 是否有交叉驗證流程                         |

#### 檢查點

**A. data-sources.md 是否包含：**

| 檢查項            | 滿分標準                  | 扣分條件        |
|-------------------|---------------------------|-----------------|
| 來源等級          | 標註官方/交易所/商業/社群 | 未標註 -5       |
| 更新頻率與延遲    | 標註 T+0/T+1/weekly       | 未標註 -5       |
| 欄位定義與單位    | 噸、盎司、合約張數等      | 缺失 -3/欄位    |
| fallback 替代方案 | 主來源掛掉怎麼辦          | 無 fallback -10 |
| 授權說明          | 資料使用授權              | 授權不明 -5     |

**B. fetch scripts 是否產出可追溯資訊：**

| 檢查項        | 滿分標準               | 扣分條件      |
|---------------|------------------------|---------------|
| 抓取時間      | 記錄 fetch timestamp   | 無記錄 -5     |
| 資料起迄      | 記錄資料的時間範圍     | 無記錄 -5     |
| 原始 endpoint | 記錄 API URL 或來源    | 無記錄 -5     |
| hash/etag     | 資料版本識別（若適用） | 無版本控制 -5 |
| cache 檔      | 有本地快取機制         | 無快取 -5     |

**C. 是否存在 cross-validate.md / explore_*：**

| 檢查項       | 滿分標準             | 扣分條件            |
|--------------|----------------------|---------------------|
| 交叉驗證流程 | 有 cross-validate.md | 無驗證流程 -10      |
| 資料探索腳本 | 有 explore_*.py      | 無探索腳本 -5       |
| 多來源比對   | 說明如何比對不同來源 | 僅單一來源無比對 -5 |

#### 評分等級

| 分數區間   | 等級   | 標準描述                                  |
|------------|--------|-------------------------------------------|
| **90-100** | 優秀   | 來源完整可追溯，有交叉驗證，fallback 完善 |
| **80-89**  | 良好   | 來源文檔完整，有快取機制，部分驗證        |
| **70-79**  | 中等   | 來源有記錄，但追溯性或驗證不足            |
| **60-69**  | 待改進 | 來源記錄不完整，無 fallback               |
| **<60**    | 不合格 | 資料來源不明或不可追溯                    |

#### 快速訊號

```
✅ 高分訊號：
  - 多個 fetch_* + explore_* + cross-validate.md：資料治理成熟度高
  - data-sources.md 有完整的 fallback 表格
  - fetch 腳本輸出有 metadata（timestamp、source、version）

⚠️ 低分訊號：
  - 只有 fetch_data.py 但 data-sources.md 很薄：「來源不穩、指標不一致」
  - 無任何 cache 機制
  - 依賴單一來源且無 fallback
```

### 2.4 穩健性與容錯（Robustness & Failure Modes）

#### 主要看哪裡

| 檔案位置                     | 評估重點                           |
|------------------------------|------------------------------------|
| `scripts/*`                  | 缺值、跳點、停牌、API error 的處理 |
| `workflows/*.md`             | 是否描述失敗模式與降級策略         |
| `references/input-schema.md` | strict/optional 欄位與預設行為     |

#### 檢查點

**A. fetch 相關：**

| 檢查項   | 滿分標準                   | 扣分條件      |
|----------|----------------------------|---------------|
| 重試機制 | 有 retry 邏輯              | 無重試 -5     |
| 限流處理 | 有 rate limit 控制         | 無限流 -5     |
| 超時設定 | 有 timeout 設定            | 無超時 -5     |
| 快取策略 | 有 cache 避免重複請求      | 無快取 -5     |
| 斷點續傳 | 大量資料支援續傳（若適用） | 該有但缺失 -5 |

**B. analyze 相關：**

| 檢查項     | 滿分標準                         | 扣分條件       |
|------------|----------------------------------|----------------|
| 缺值處理   | 明確 drop/ffill/interpolate 策略 | 未處理 -10     |
| 極端值處理 | winsorize/clip 或明確標示        | 未處理 -5      |
| 資料不足   | window 不夠時回報，不硬算        | 硬算 -10       |
| 錯誤邊界   | try-except 適當包裹              | 無錯誤處理 -10 |

**C. output 相關：**

| 檢查項   | 滿分標準                     | 扣分條件      |
|----------|------------------------------|---------------|
| 安全輸出 | 主路徑失敗時有降級輸出       | 直接崩潰 -10  |
| 錯誤回報 | 說明缺什麼、怎麼補、替代方案 | 僅錯誤代碼 -5 |
| 部分結果 | 可輸出部分完成的結果         | 全有全無 -5   |

#### 評分等級

| 分數區間   | 等級   | 標準描述                           |
|------------|--------|------------------------------------|
| **90-100** | 優秀   | 完善的錯誤處理、降級策略、安全輸出 |
| **80-89**  | 良好   | 主要錯誤有處理，有基本降級         |
| **70-79**  | 中等   | 部分錯誤處理，降級不完整           |
| **60-69**  | 待改進 | 錯誤處理零散，容易崩潰             |
| **<60**    | 不合格 | 無錯誤處理，極易失敗               |

#### 快速訊號

```
✅ 高分訊號：
  - 有 workflows/monitor.md：代表會反覆跑，更需要穩健性
  - 有 scripts/explore_*.py：把資料異常當常態處理
  - references/failure-modes.md 存在且完整

⚠️ 低分訊號：
  - scripts 無 try-except
  - 缺值直接導致程式崩潰
  - 無任何降級策略描述
```

### 2.5 可重現性與可維護性（Reproducibility & Maintainability）

#### 主要看哪裡

| 檔案位置                       | 評估重點                                  |
|--------------------------------|-------------------------------------------|
| `manifest.json` / `skill.yaml` | 版本、依賴、工具介面、參數定義            |
| `templates/output-json.md`     | 輸出 schema 是否穩定（利於下游 pipeline） |
| `scripts/`                     | 模組化程度、參數集中、避免 magic numbers  |
| `examples/`                    | 回歸測試素材                              |

#### 檢查點

**A. skill.yaml / manifest.json 是否：**

| 檢查項                     | 滿分標準                              | 扣分條件        |
|----------------------------|---------------------------------------|-----------------|
| semantic version           | 有 semver 版本號                      | 無版本 -10      |
| inputs/outputs schema 版本 | 明確列出                              | 缺失 -5         |
| 參數預設值                 | 有預設且與 methodology 一致           | 缺失或不一致 -5 |
| 依賴清單                   | requirements.txt 或 dependencies 完整 | 缺失 -5         |

**B. templates/ 是否穩定：**

| 檢查項       | 滿分標準         | 扣分條件     |
|--------------|------------------|--------------|
| 欄位命名固定 | 欄位名不隨版本變 | 不穩定 -10   |
| 類型固定     | 欄位類型不變     | 類型飄移 -10 |
| 向後相容     | 新版本相容舊輸出 | 破壞相容 -10 |

**C. scripts/ 是否避免散落 magic numbers：**

| 檢查項   | 滿分標準                  | 扣分條件    |
|----------|---------------------------|-------------|
| 參數集中 | threshold/window 在配置區 | 分散 -3/處  |
| 模組化   | 功能分離，單一職責        | 功能混合 -5 |
| 可配置   | 關鍵參數可外部傳入        | 硬編碼 -5   |

**D. 文件漂移檢查：**

| 檢查項     | 滿分標準                              | 扣分條件    |
|------------|---------------------------------------|-------------|
| 無重複文件 | 不存在 root + references/ 都有同名 md | 有重複 -10  |
| 單一來源   | 每個概念只有一處定義                  | 多處定義 -5 |

#### 評分等級

| 分數區間   | 等級   | 標準描述                           |
|------------|--------|------------------------------------|
| **90-100** | 優秀   | 完整版本管理，模板穩定，無文件漂移 |
| **80-89**  | 良好   | 版本清楚，模板大致穩定             |
| **70-79**  | 中等   | 基本可維護，但有散落參數           |
| **60-69**  | 待改進 | 維護成本高，有文件漂移風險         |
| **<60**    | 不合格 | 難以維護，版本混亂                 |

#### 快速訊號

```
✅ 高分訊號：
  - 有 templates/output-json.md + examples/sample_output.json：適合做 contract test
  - manifest.json 有完整的 dependencies 和 version
  - 所有 threshold/window 在 CONFIG 區集中定義

⚠️ 低分訊號：
  - 同 repo 出現兩份 methodology（root + references/）：文件漂移
  - scripts 中 magic numbers 分散各處
  - 無 version 欄位
```

### 2.6 輸出可用性與決策支援（Usability & Decision Support）

#### 主要看哪裡

| 檔案位置                                           | 評估重點                           |
|----------------------------------------------------|------------------------------------|
| `templates/output-markdown.md` / `output-json.md`  | 對人友善 + 對機器友善              |
| `scripts/visualizer.py` / `generate_chart.py`      | 是否提供直覺圖像與摘要             |
| `workflows/visualize.md`、`historical-episodes.md` | 是否提供「如何解讀」與「歷史對照」 |

#### 檢查點

**A. Markdown 模板是否包含：**

| 檢查項              | 滿分標準                      | 扣分條件 |
|---------------------|-------------------------------|----------|
| TL;DR（結論一句話） | 開頭有一句話結論              | 缺失 -10 |
| 依據                | 最重要 3 個數據/圖支撐結論    | 缺失 -10 |
| 假設與風險          | 資料延遲、指標適用期          | 缺失 -5  |
| 下一步              | 監控、條件觸發、對沖/避險提示 | 缺失 -5  |

**B. JSON 模板是否包含（建議欄位）：**

| 欄位          | 說明          | 扣分條件 |
|---------------|---------------|----------|
| `status`      | 執行狀態      | 缺失 -5  |
| `confidence`  | 信心水準      | 缺失 -5  |
| `reasons[]`   | 結論依據清單  | 缺失 -5  |
| `artifacts[]` | 圖表路徑/數據 | 缺失 -3  |

**C. 是否有歷史對照與可視化：**

| 檢查項              | 滿分標準           | 扣分條件         |
|---------------------|--------------------|------------------|
| historical-episodes | 有歷史事件對照     | 缺失 -10         |
| backtest/compare    | 有回測或比較功能   | 該有但缺失 -5    |
| visualizer 腳本     | 有圖表生成         | 缺失 -5          |
| 解讀說明            | 有「如何解讀」文檔 | 只有圖無解讀 -10 |

#### 評分等級

| 分數區間   | 等級   | 標準描述                               |
|------------|--------|----------------------------------------|
| **90-100** | 優秀   | 輸出完整可用，有歷史對照，決策指引清晰 |
| **80-89**  | 良好   | 輸出結構完整，有基本可視化             |
| **70-79**  | 中等   | 輸出可用，但缺乏解讀或歷史對照         |
| **60-69**  | 待改進 | 輸出格式不完整，缺乏決策指引           |
| **<60**    | 不合格 | 輸出難以使用，無法支持決策             |

#### 快速訊號

```
✅ 高分訊號：
  - 有 historical-episodes.md：把現在放到歷史分布裡，很加分
  - output-markdown.md 有 TL;DR + 依據 + 風險 + 下一步
  - 有 visualizer.py 且搭配解讀模板

⚠️ 低分訊號：
  - 只有圖、沒有解讀模板：「看得到圖，但不知道怎麼用」
  - JSON 輸出只有 data，無 status/confidence
  - 無任何歷史對照或 backtest
```

## 4. Repo QA 一頁式清單

對每個 Skill repo 跑一遍：

### 快速檢查

| 維度               | 核心問題                                                             | 快速判斷                           |
|--------------------|----------------------------------------------------------------------|------------------------------------|
| **Fit**            | `SKILL.md` + `workflows/` 是否閉環？`input-schema` 是否清晰？        | 有 analyze.md + output 模板 = 閉環 |
| **Correctness**    | `methodology` 能否重現？`scripts` 是否吻合？`examples` 能否回歸？    | 有公式 + 有範例 = 可驗證           |
| **Data**           | `data-sources` 是否完整？fetch 是否可追溯？是否有 `cross-validate`？ | 有 fallback + 有 cache = 治理完善  |
| **Robustness**     | 失敗模式是否寫明？`scripts` 是否做降級/錯誤回報？                    | 有 try-except + 有降級 = 穩健      |
| **Repro/Maintain** | 版本/模板/schema 是否穩定？是否避免文件漂移？                        | 有 version + 無重複 md = 可維護    |
| **Usability**      | 模板是否含結論/依據/風險/下一步？是否有歷史對照與可視化？            | 有 TL;DR + 有 historical = 可用    |

### 詳細清單

**任務適配度與問題定義（Problem Fit）**
- [ ] SKILL.md 有一句話目標
- [ ] SKILL.md 說明適用/不適用情境
- [ ] workflows/ 有完整主路徑
- [ ] input-schema.md 欄位定義清晰

**正確性與可驗證性（Correctness）**
- [ ] methodology.md 有可重現的公式/規則
- [ ] scripts/ 與 methodology 吻合
- [ ] examples/ 有 golden case 可回歸

**資料來源品質與資料治理（Data Governance）**
- [ ] data-sources.md 完整（來源、頻率、單位、fallback）
- [ ] fetch scripts 有 cache 和 timestamp
- [ ] 有 cross-validate 或 explore 流程

**穩健性與容錯（Robustness）**
- [ ] scripts 有適當的 try-except
- [ ] 有缺值/極端值處理策略
- [ ] 失敗時有降級輸出或錯誤回報

**可重現性與可維護性（Maintainability）**
- [ ] manifest.json 有 semver 版本
- [ ] templates/ 欄位命名穩定
- [ ] 無重複文件（root 和 references/ 不同時有同名 md）

**輸出可用性與決策支援（Usability）**
- [ ] output-markdown.md 有 TL;DR + 依據 + 風險 + 下一步
- [ ] output-json.md 有 status + confidence + reasons
- [ ] 有歷史對照或可視化

## 5. skill.yaml 配置範例

### 5.1 完整範例

```yaml
qualityScore:
  overall: 70
  badge: 白銀
  metrics:
    problemFit: 75
    correctness: 80
    dataGovernance: 65
    robustness: 65
    maintainability: 70
    usability: 75
  details: |
    **任務適配度（75/100）**
    - SKILL.md 有清晰的一句話目標
    - workflows/analyze.md 覆蓋主路徑
    - input-schema.md 欄位定義大致清楚
    - 缺少不適用情境說明

    **正確性（80/100）**
    - methodology.md 有完整公式推導
    - scripts/ 實作與文檔吻合
    - examples/ 有 1 個 golden case
    - 可增加更多邊界案例

    **資料治理（65/100）**
    - data-sources.md 有來源清單
    - 缺少 fallback 替代方案
    - fetch 有 cache 但無 timestamp 記錄

    **穩健性（65/100）**
    - 有基本的錯誤處理
    - 缺值處理策略不明確
    - 無降級輸出機制

    **可維護性（70/100）**
    - manifest.json 有 version
    - 部分 magic numbers 分散
    - 無重複文件

    **輸出可用性（75/100）**
    - output-markdown.md 有 TL;DR 和依據
    - 缺少下一步建議
    - 有基本可視化，缺歷史對照
```
